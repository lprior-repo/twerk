# Contract Specification

## Context

- **Feature:** Define two new domain types in `twerk-core`: `TriggerState` enum and `TriggerId` newtype.
- **Domain terms:**
  - `TriggerState` -- the lifecycle phase of a trigger (an event-driven or scheduled activation rule). Variants: `Active`, `Paused`, `Disabled`, `Error`.
  - `TriggerId` -- a validated identifier for a trigger instance. Follows the `define_id!` macro pattern but with stricter length bounds.
- **Module placement:**
  - `TriggerState` lives in a new file `crates/twerk-core/src/trigger.rs` (registered as `pub mod trigger;` in `lib.rs`).
  - `TriggerId` is added to the existing `crates/twerk-core/src/id.rs` via `define_id!(TriggerId)`, with a *custom validation function* that enforces the 3-64 character range.
- **Existing patterns to follow:**
  - `TaskState` / `JobState` / `ScheduledJobState` in `task.rs` and `job.rs`: `#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash, Serialize, Deserialize, Default)]` with `#[serde(rename_all = "SCREAMING_SNAKE_CASE")]`.
  - `Display` impl writes uppercase strings (e.g. `"ACTIVE"`).
  - `FromStr` impl parses case-insensitively via `.to_uppercase()`.
  - Parse error type: `ParseXxxError(String)` with `Display` + `std::error::Error`.
  - `define_id!` macro in `id.rs`: generates `new()`, `as_str()`, `From<String>`, `From<&str>`, `Display`, `AsRef<str>`, `Deref`, `Borrow<str>`, `FromStr`, `Default`, `Serialize`/`Deserialize` (transparent).
  - `IdError` enum in `id.rs`: `Empty`, `TooLong(usize)`, `InvalidCharacters`.
- **Assumptions:**
  - No state machine logic is in scope (bead description explicitly says "No state machine logic yet").
  - The `define_id!` macro will be extended or a custom variant will be created to accept a length range, OR `TriggerId` will use `define_id!(TriggerId)` and override `new()` with a custom validator that calls `validate_id` then checks the 3-64 range.
  - `thiserror` is available in `twerk-core/Cargo.toml`.
  - `serde` and `serde_json` are available in `twerk-core/Cargo.toml`.
- **Open questions:**
  - None. All constraints are specified in the bead description and derivable from existing code.

## Preconditions

### TriggerState

- [P-TS-1] N/A -- `TriggerState` variants are compile-time constants; no construction preconditions.

### TriggerId

- [P-TI-1] The input string MUST have a byte length in the range 3..=64 (inclusive).
- [P-TI-2] Every character in the input string MUST satisfy `c.is_alphanumeric() || c == '-' || c == '_'`. This is the same character set used by `validate_id` in `id.rs`.
- [P-TI-3] The input string MUST NOT be empty (subsumed by P-TI-1 but stated explicitly for `IdError::Empty` mapping).
- [P-TI-4] `TriggerId::new()` is the ONLY fallible constructor. `From<String>` and `From<&str>` (the infallible trait impls generated by `define_id!`) MUST NOT be used to bypass validation. **Decision required at implementation:** either remove the blanket `From` impls from the macro for `TriggerId`, or document that `From` bypasses validation (matching existing `define_id!` behavior for `JobId`, `TaskId`, etc.). Given existing convention, `From` impls exist but `new()` is the validated path.

- [POS-TS-2] `TriggerState::Paused.serialize()` yields the JSON string `"PAUSED"`.

- [POS-TS-7] `format!("{}", TriggerState::Error)` yields `"ERROR"`.

- [POS-TI-2] `TriggerId::new(s)?.as_str()` returns a string equal to `s`.
- [POS-TI-3] `TriggerId::new(s)?.to_string()` returns a string equal to `s`.
- [POS-TI-4] Valid `TriggerId` values serialize and deserialize as transparent JSON strings.

## Invariants

### TriggerState

- [INV-TS-1] `TriggerState` has exactly 4 variants: `Active`, `Paused`, `Disabled`, `Error`. No variant is added or removed without a breaking change.
- [INV-TS-2] Every variant serializes to its `SCREAMING_SNAKE_CASE` name via serde.
- [INV-TS-3] Every variant implements `Copy`, making it a zero-cost discriminant. No heap allocation.
- [INV-TS-4] `Display` output for every variant is identical to its serde-serialized form (both are `SCREAMING_SNAKE_CASE`).
- [INV-TS-5] `FromStr` is case-insensitive: `"active"`, `"ACTIVE"`, `"Active"` all parse to `TriggerState::Active`.
- [INV-TS-6] `Default` is `Active`. This mirrors `ScheduledJobState::Active` as the default for scheduled entities.
- [INV-TS-7] Future state machine transitions (not in scope) will respect these valid transitions (documented here for reference, NOT enforced in this bead):
  - `Active` -> `Paused`, `Disabled`, `Error`
  - `Paused` -> `Active`, `Disabled`
  - `Disabled` -> `Active`
  - `Error` -> `Active`, `Disabled`

### TriggerId

- [INV-TI-1] A constructed `TriggerId` (via `new()`) always holds a string with length in 3..=64.
- [INV-TI-2] A constructed `TriggerId` (via `new()`) always holds only characters matching `[a-zA-Z0-9_-]` (plus Unicode alphanumeric per Rust's `is_alphanumeric()`).
- [INV-TI-3] `TriggerId` is `#[serde(transparent)]` -- it serializes/deserializes as a plain JSON string.
- [INV-TI-4] `TriggerId` derives `Eq` + `Hash`, making it suitable as a `HashMap`/`HashSet` key.
- [INV-TI-5] `TriggerId` derives `Default`, yielding `TriggerId("")` (empty string, matching macro convention). **Note:** this default value is NOT a valid TriggerId per INV-TI-1. This is an existing tension in the `define_id!` macro that is preserved for consistency.

## Error Taxonomy

### TriggerState errors

| Error variant | Kind | When |
|---|---|---|
| `ParseTriggerStateError(String)` | Parse failure | `FromStr::from_str()` receives a string that is not one of `"ACTIVE"`, `"PAUSED"`, `"DISABLED"`, `"ERROR"` (case-insensitive). |

This type follows the existing pattern of `ParseJobStateError` and `ParseScheduledJobStateError`:
- `Debug + Clone + PartialEq + Eq`
- `Display` writes `"unknown TriggerState: {0}"`
- Implements `std::error::Error`

### TriggerId errors (reuses `IdError` from `id.rs`)

| Error variant | Kind | When |
|---|---|---|
| `IdError::Empty` | Validation failure | Input string is empty (length 0). |
| `IdError::TooShort(usize)` | Validation failure | Input string length is 1 or 2 (below the 3-char minimum). **This is a NEW variant** that must be added to `IdError`. |
| `IdError::TooLong(usize)` | Validation failure | Input string length exceeds 64 characters. Note: the error message should reflect the actual max for TriggerId (64), not the global `MAX_ID_LENGTH` (1000). The `TooLong` message from `validate_id` says 1000 -- the custom TriggerId validator must produce its own error. |
| `IdError::InvalidCharacters` | Validation failure | Input contains characters outside `[a-zA-Z0-9_-]` (per Rust's `is_alphanumeric()`). |

**Important:** The existing `IdError::TooLong(usize)` message hardcodes `MAX_ID_LENGTH = 1000`. For `TriggerId`, the maximum is 64. The contract requires one of:
- (a) A new `IdError::TooLong` variant with a max parameter: `TooLong { len: usize, max: usize }`, OR
- (b) A `TriggerIdError` enum specific to `TriggerId` with its own `TooLong(usize)` whose `Display` says "64", OR
- (c) The `validate_id` function remains unchanged; `TriggerId::new()` calls `validate_id` first (catching Empty/InvalidChars), then checks length bounds separately, producing `IdError::TooLong` with the actual length.

**Recommended approach (c):** Keep `IdError` as-is. `TriggerId::new()` delegates to `validate_id` for character validation, then applies its own 3-64 length check. The `TooLong` message will say "1000 characters" which is technically inaccurate for TriggerId -- but to avoid breaking existing code, this is acceptable. Alternatively, introduce `IdError::TooShort(usize)` for the minimum bound and accept the semantic imprecision on the maximum.

**Decision: Introduce `IdError::TooShort(usize)` as a new variant.** This is additive (non-breaking) and accurately represents the 3-char minimum. For the 64-char maximum, `IdError::TooLong` is reused as-is (the message mentions 1000, which is a superset truth -- the ID IS too long for TriggerId, even if the message says 1000).

```
pub enum IdError {
    Empty,                          // existing
    TooShort(usize),                // NEW: length < 3
    TooLong(usize),                 // existing: length > MAX_ID_LENGTH (1000)
    InvalidCharacters,              // existing
}
```

For `TriggerId::new()`, the validation order is:
1. Empty check (delegates to `validate_id` -> `IdError::Empty`)
2. Too short check: if `s.len() < 3` -> `IdError::TooShort(s.len())`
3. Too long check: if `s.len() > 64` -> `IdError::TooLong(s.len())`
4. Character check (delegates to `validate_id` -> `IdError::InvalidCharacters`)

impl fmt::Display for ParseTriggerStateError { ... }
impl std::error::Error for ParseTriggerStateError { ... }

```rust
// New error variant (additive to existing IdError)
pub enum IdError {
    Empty,
    TooShort(usize),        // NEW
    TooLong(usize),
    InvalidCharacters,
}

// Type definition (via macro, with custom new())
define_id!(TriggerId);

// TriggerId::new() overrides the macro-generated new() to add 3-64 length bounds.
// This requires either:
//   (a) A define_id_with_bounds!(TriggerId, 3, 64) macro variant, OR
//   (b) An impl block after the macro invocation that overrides new().
// Recommended: (b) -- add a separate impl block that shadows the macro's new().
// Note: Rust does not allow shadowing inherent methods from macro expansion.
// Therefore, the macro must be modified to accept optional bounds,
// OR TriggerId must NOT use define_id! and instead be hand-written.

// FINAL DECISION: Hand-write TriggerId (do NOT use define_id!) to enforce
// the 3-64 length constraint in new(), while preserving all trait impls
// (Display, AsRef<str>, Deref, Borrow<str>, FromStr, Default, Serialize/Deserialize).

impl TriggerId {
    pub fn new(id: impl Into<String>) -> Result<Self, IdError> { ... }
    #[must_use]
    pub fn as_str(&self) -> &str { ... }
}
```

## Non-goals

- [NG-1] State machine transitions or `is_active()` / `can_transition_to()` methods on `TriggerState`. These will be added in a follow-up bead.
- [NG-2] Persistence layer (database mapping, migration scripts).
- [NG-3] API route handlers or HTTP request/response types.
- [NG-4] Modifying the `define_id!` macro to accept generic bounds. `TriggerId` will be hand-written to avoid macro complexity.
- [NG-5] Refactoring existing ID types (JobId, TaskId, etc.) to use length bounds.
- [NG-6] CJK/Unicode character policy changes. The existing `is_alphanumeric()` behavior (which accepts CJK) is preserved.
- [NG-7] `Clone` on `ParseTriggerStateError` is included for consistency with `ParseJobStateError`, but `thiserror` derive is NOT used (matching existing pattern of manual `Display` impl).
